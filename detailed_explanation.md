# MathorCup C题 详细解析与复现说明

本文件仅用于实现说明、复现实录和结果追溯，不作为正式参赛论文提交件；正式定稿以 `solution_report.md` 和 `example.tex` 为准。

## 1. 文件结构

当前项目的核心交付物如下：

| 文件 | 作用 |
|---|---|
| `src/pyCode/common.py` | 统一数据路径、列名、阈值、成本参数、输出目录、图表配置 |
| `src/pyCode/problem1.py` | 问题 1：双层指标筛选与九种体质贡献分析 |
| `src/pyCode/problem2.py` | 问题 2：双轨风险预警模型与三级风险分层 |
| `src/pyCode/problem3.py` | 问题 3：动态规划版 6 个月干预优化 |
| `solution_report.md` | Markdown 主论文 |
| `example.tex` | LaTeX 正式排版稿 |
| `detailed_explanation.md` | 本说明文档 |
| `src/Data` | 赛题样例数据 |
| `src/outputs/figures` | 所有图形结果 |
| `src/outputs/tables` | 所有表格、汇总表、JSON 摘要 |

## 2. 运行方式

在项目根目录执行：

```powershell
python src/pyCode/problem1.py
python src/pyCode/problem2.py
python src/pyCode/problem3.py
```

脚本会自动从 `src/Data/附件1：样例数据.xlsx` 读取样例数据，不再依赖 Linux 硬编码路径。所有输出统一进入 `src/outputs` 目录。若 Windows 终端默认编码异常，可在命令前额外加上 `-X utf8`。

## 3. 公共模块 common.py 说明

`common.py` 的设计目标是让三问共享完全一致的数据口径。

### 3.1 数据与列名

`Columns` 数据类统一定义全部 37 个字段名称，避免脚本各自手写中文列名带来的错字和编码问题。

### 3.2 阈值口径

公共模块中维护了以下口径：

1. 血脂正常范围：
   - TC：3.1-6.2
   - TG：0.56-1.7
   - LDL-C：2.07-3.1
   - HDL-C：1.04-1.55
2. 代谢正常范围：
   - 空腹血糖：3.9-6.1
   - BMI：18.5-23.9
   - 血尿酸：男 208-428，女 155-357
3. 干预参数：
   - 中医等级成本：30 / 80 / 130 元每月
   - 活动强度成本：3 / 5 / 8 元每次
   - 活动时长：10 / 20 / 30 分钟每次

此外，`common.py` 还统一定义了问题 1 和问题 2 共用的特征分组规则：

1. `ACTIVITY_ITEM_FEATURES`：10 个 ADL/IADL 单项，只供线性模型、L1/LASSO 和 VIF 使用。
2. `ACTIVITY_SUMMARY_FEATURES`：`ADL总分`、`IADL总分`、`活动量表总分`，只用于描述性统计和结构性共线性审计。
3. `TREE_ACTIVITY_FEATURES`：10 个活动单项加 `活动量表总分`，供树模型使用，但明确删除 `ADL总分` 和 `IADL总分`。
4. `LINEAR_EARLY_SCREENING_FEATURES`：九种体质积分 + 10 个活动单项 + 代谢指标 + 人口学变量。
5. `TREE_EARLY_SCREENING_FEATURES`：在线性口径基础上增加 `活动量表总分`，形成树模型输入。

### 3.3 输出管理

所有图表都通过 `figure_path()` 输出到 `src/outputs/figures`；所有表格通过 `save_table()` 或 `save_workbook()` 输出到 `src/outputs/tables`。这样主论文和 `example.tex` 只需要引用 `src/outputs` 下的稳定产物。

## 4. 问题 1 实现细节

## 4.1 为什么先做“变量处理原则”

这次重构里，问题 1 不再把 LASSO 结果直接当结论，而是先回答三个变量处理问题：

1. “因变量多重共线性”这个说法本身不成立，真正要检查的是自变量共线性。
2. `活动量表总分 = ADL总分 + IADL总分`，而 `ADL总分`、`IADL总分` 又分别等于各自单项之和，所以在线性模型里同时放“单项 + 总分”必然产生结构性共线性。
3. 当目标是“高血脂症二分类标签”时，`TC/TG/LDL-C/HDL-C` 其实就是标签定义本身，必须从预测型模型中删除，否则会形成标签泄漏。

所以问题 1 的代码先定规则，再做建模，而不是反过来用模型结果替代数据逻辑。

## 4.2 共线性审计

`problem1.py` 中新增了独立的共线性审计模块，输出四张表：

1. `problem1_activity_vif_raw`：原始活动变量 VIF，预期 `inf`。
2. `problem1_activity_vif_clean`：删掉活动总分族变量后的 VIF。
3. `problem1_constitution_vif`：九种体质积分的 VIF。
4. `problem1_constitution_corr_pairs`：九种体质两两相关系数排序。

这一步回答了两个争议：

1. 真正严重的共线性来自活动总分体系，而不是九种体质。
2. 九种体质的 VIF 约 1.02-1.21，最大相关仅约 0.223，因此没有必要因为“体质间共线性”删变量。

## 4.3 双层主分析

问题 1 的主分析被拆成两个层次：

1. 直接诊断层：只看 `TC/TG/LDL-C/HDL-C`，用于说明哪些变量与题目标签最一致。
2. 辅助预警层：只在“非直接血脂变量 + 清洗后的活动单项”上做加权 Borda 排序。

其中 `compute_feature_statistics()` 对每个候选指标统一计算：

1. 确诊组和未确诊组均值。
2. 正态性检验，并自动选择 Welch t 或 Mann-Whitney U。
3. Cohen's $d$。
4. 与痰湿积分的 Pearson、Spearman 相关。
5. 单指标 ROC AUC 增益。
6. 互信息。
7. FDR 校正后的显著性。

辅助预警层的综合分由三部分组成：

1. 痰湿解释度 40%。
2. 非直接预警能力 40%。
3. Bootstrap 稳定性 20%。

代码中通过加权 Borda 合成，而不是简单平均，这样更适合把“相关、差异、稳定性”三种不同口径拼到一张综合榜上。

## 4.4 L1-Logit 与 Lasso 稳健性检验

为了回应“LASSO 为什么没选出很多变量”的质疑，代码把 L1/LASSO 改成了明确的对照实验：

1. 诊断目标下，`run_l1_logit_experiment()` 比较四种配置：
   - 标准化 + 去总分 + 去直接血脂
   - 标准化 + 去总分 + 保留直接血脂
   - 标准化 + 保留总分 + 去直接血脂
   - 不标准化 + 去总分 + 去直接血脂
2. 痰湿目标下，`run_lasso_experiment()` 只比较两种配置：
   - 标准化 + 去总分 + 含血脂
   - 标准化 + 去总分 + 不含血脂

这样做的目的不是“非要让 LASSO 选出变量”，而是验证：

1. 正确口径下，诊断型 L1-Logit 是否只剩少量非泄漏变量。
2. 一旦把直接血脂放回去，AUC 是否会异常升高。
3. 痰湿目标是否根本不适合线性稀疏拟合。

当前结果正是这三点：正式口径只稳定选出血尿酸，加入直接血脂后 AUC 跃升，而痰湿型 `LassoCV` 基本收缩到全零解。

## 4.5 九种体质贡献分析

代码没有直接用 `sklearn` 的黑箱 Logistic，而是使用 `statsmodels.GLM(Binomial)` 输出：

1. 回归系数。
2. OR。
3. 95% 置信区间。
4. p 值。

同时模型中只放九种体质积分，不把“体质标签”再并入同一模型，从而避免语义重复。这样论文才能给出规范的医学统计表达，并且明确区分“体质标签患病率”与“体质积分调整后贡献”。

## 5. 问题 2 实现细节

## 5.1 为什么必须拆成双轨模型

在正式实现前，代码先用 `verify_label_consistency()` 检查：

```text
标签 == (TC高 或 TG高 或 LDL高 或 HDL低)
```

结果一致率为 1.0000。也就是说，如果把这四项直接扔进分类器，模型高分只是因为它学到了标签定义本身。

因此，问题 2 必须拆成：

1. 轨道 A：无泄漏早筛模型
2. 轨道 B：临床风险分层模型

## 5.2 轨道 A 的实现

轨道 A 延续问题 1 的变量清洗策略，但现在改成“两套特征口径 + 两层评估流程”。

特征口径如下：

1. 线性模型使用 `LINEAR_EARLY_SCREENING_FEATURES`：
   - 九种体质积分
   - 10 个 ADL/IADL 单项
   - 空腹血糖、血尿酸、BMI
   - 年龄组、性别、吸烟史、饮酒史
2. 树模型使用 `TREE_EARLY_SCREENING_FEATURES`：
   - 在线性口径基础上保留 `活动量表总分`
   - 但明确不引入 `ADL总分`、`IADL总分`

候选模型扩展为五类：

1. Logistic 回归
2. 随机森林
3. 梯度提升树
4. 极端随机树
5. 直方梯度提升树

评估流程分成两层：

1. 第一层：`RepeatedStratifiedKFold(5×5)` 比较模型家族，记录每折 AUC、PR-AUC、Brier、F1。
2. 第二层：对入选家族再做“原始版 vs Sigmoid 校准版”比较，并用固定 5 折 OOF 生成最终 `p_early`。

这样做的目的，是把“哪个模型家族最稳”和“这个家族是否值得校准”两个问题分开回答。

## 5.3 为什么最终选“直方梯度提升树 + Sigmoid 校准”

从结果上看：

1. Logistic 只能捕捉线性关系，表现最弱。
2. 随机森林和极端随机树都能捕捉非线性，但 AUC 和 Brier 都落后于两类 boosting 模型。
3. 直方梯度提升树在 repeated-CV 下取得最高的平均 AUC 和最低的平均 Brier，且标准差也最小，因此先成为最佳模型家族。
4. 在其基础上做 `Sigmoid` 校准后，AUC 从 0.8946 升到 0.8994，Brier 从 0.1027 降到 0.1018，所以最终模型升级为“直方梯度提升树 + Sigmoid 校准”。

这和题目的变量结构一致：体质、活动、代谢之间存在明显的非线性交互。更重要的是，轨道 A 的高分不再来自“把标签定义重新学一遍”，而是来自清洗后变量中的真实非线性信号。

代码随后又补了三层证据：

1. 固定 5 折 OOF 的 Bootstrap 置信区间。
2. 特征块消融实验。
3. 按性别、年龄组、痰湿/非痰湿体质的亚组稳健性分析。

因此，问题 2 不再只是“一个模型表”，而是一整条完整的模型证据链。

## 5.4 轨道 B 风险分构成

轨道 B 中的连续风险分由五部分构成：

### 5.4.1 `lipid_burden`

用于刻画血脂异常项数以及异常方向：

1. TC/TG/LDL 升高记为“高脂方向”
2. HDL 降低记为“保护因子下降方向”
3. 异常项数越多、方向越复杂，负担越大

### 5.4.2 `lipid_excess`

并不只看“是否异常”，而是看超阈值幅度。代码中对每一项都计算超阈比例，再把总和归一化到 0-1。

### 5.4.3 `early_risk`

直接使用轨道 A 输出的 OOF 概率：

$$
early\_risk = p_{early}
$$

也就是说，轨道 B 不会丢掉轨道 A 中的无泄漏早筛信息。

### 5.4.4 `metabolic_modifier`

根据空腹血糖、尿酸和 BMI 的异常项数计算，再归一化到 0-1。

### 5.4.5 `function_modifier`

由痰湿积分和活动总分共同决定，体现“痰湿偏颇 + 功能受限”的复合风险。

五个分量统一归一化后，按如下基准权重线性加权：

1. `lipid_burden`：0.32
2. `lipid_excess`：0.22
3. `early_risk`：0.19
4. `metabolic_modifier`：0.12
5. `function_modifier`：0.15

这一步的意义在于：风险分仍然可解释，但不再是“手写一串分值”，而是“可重复构造的归一化线性分层”。

## 5.5 阈值搜索逻辑

代码不会手工指定低、中、高分界点，而是对候选阈值做穷举搜索，要求：

1. 三组样本量不能太小
2. 三组确诊率必须严格递增
3. 组间分离度最大

这一步的好处是：

1. 结论可复现
2. 阈值选择有客观依据
3. 不会出现“为了好看强行定阈值”的问题

在冲奖版里，阈值搜索又补了两层稳健性：

1. 对 200 次 Bootstrap 重采样重新搜索 `(T_low, T_high)`，输出阈值均值、标准差和 95% 区间。
2. 对五个风险分量逐一做 `±20%` 权重扰动，检查三级患病率是否仍保持单调。

这样阈值不只是“算出来一个值”，而是“证明这个值附近都稳定”。

## 5.6 痰湿高风险规则提取

手工列规则有两个问题：

1. 容易主观挑选
2. 容易和前面的风险分定义脱节

因此代码不再只训练一棵树，而是在痰湿体质子样本上做 100 次 Bootstrap。每次都训练一棵 `DecisionTreeClassifier(max_depth=3, min_samples_leaf=12)`，然后：

1. 提取所有高风险叶节点规则。
2. 把逻辑等价规则做规范化和合并。
3. 在完整痰湿子样本上重新评估每条规则的支持度、高风险纯度和确诊率。
4. 最终只保留“出现频率足够高、支持样本数不少于 10、纯度较高”的稳定规则。

这样得到的规则既来源于数据，也足够可解释，而且不会因为某一次抽样偶然切出了某个阈值就被直接写进正文。

## 6. 问题 3 实现细节

## 6.1 为什么从穷举升级到动态规划

原始思路默认“6 个月固定一套活动计划 + 固定中医等级”，这会忽略一个非常重要的事实：

1. 痰湿积分会下降；
2. 一旦积分下降到阈值以下，中医等级就应该随之下降；
3. 活动方案也可能在预算边界附近发生更优切换。

所以这里把问题建模成动态规划而不是一次性穷举。

## 6.2 状态、动作和目标

状态由以下量隐含决定：

1. 当前月份
2. 当前痰湿积分
3. 已花费预算
4. 上个月的活动强度与频率

动作是当月选择的：

1. 活动强度
2. 每周频率

中医等级不是自由动作，而是由当前积分自动映射得到。

目标采用字典序：

1. 先最小化第 6 月末积分
2. 再最小化总成本
3. 最后最小化切换次数

## 6.3 为什么最终用了“前沿剪枝”而不是递归缓存

最初的递归 DP 把预算维度展开得太大，278 位患者批量求解会超时。最终实现改为：

1. 按月份逐层展开状态前沿
2. 只保留同一 `(月末积分, 本月动作)` 下成本和切换次数更优的状态
3. 在第 6 月末统一选取字典序最优解

这个版本既是精确最优，又能稳定在几十秒内求完整个痰湿子样本。

## 6.4 为什么只需要 31 个模板

对本题数据来说，动态规划模板实际上只与两个量有关：

1. 初始痰湿积分
2. 最大可选活动强度

痰湿体质患者的初始积分只取 55-65 这 11 个整数，而最大可选强度只可能是 1、2、3，所以最终只有 31 组模板。代码先求模板，再映射回 278 个具体患者，这大幅提高了求解效率。

## 6.5 为什么还要加“固定活动基线”

如果只给出动态规划结果，论文很容易被问到一个关键问题：

“动态规划到底是真有价值，还是只是把固定方案换了个更复杂的写法？”

所以这轮代码专门补了一条固定活动基线：

1. 活动强度和训练频率在 6 个月内保持不变。
2. 中医等级仍然随当前痰湿积分自动下调。
3. 仍然使用和动态规划完全相同的预算约束与评分规则。

这样做以后，动态与静态的差异就只剩下一个来源：

“活动方案是否允许按月切换。”

结果非常清楚：

1. 最大强度为 1 的 92 人，动态和静态完全一致。
2. 最大强度为 2 的 171 人，动态和静态也完全一致。
3. 只有最大强度为 3 的 15 人，动态规划才真正带来平均 1.4007 分的终点改善。

这反而让问题 3 更有说服力，因为它说明模型没有把所有人都“强行动态化”，而是识别出了真正需要动态切换的边界患者。

## 6.6 为什么要做预算敏感性和可行性审计

预算 2000 元是题目硬约束，但在论文里我们还希望回答两个更深的问题：

1. 这个预算是否真的决定了方案质量？
2. 代码实现是否严格遵守了题目约束？

所以脚本又补了两层验证。

第一层是预算敏感性：

1. 把预算上限分别设为 1200、1400、1600、1800、2000 元。
2. 对每个预算重新求动态最优。
3. 比较总体以及不同最大强度组的平均终点积分、平均成本和切换率。

这样可以看出：

1. 最大强度为 1 的患者在约 1000 元预算就已达到平台。
2. 最大强度为 2 的患者在 1600 元左右基本达到平台。
3. 只有最大强度为 3 的患者会持续受益于 1800-2000 元预算。

第二层是可行性审计：

1. 检查是否存在预算超支。
2. 检查月度积分是否出现反弹。
3. 检查活动强度是否越过年龄/量表上限。
4. 检查模板复用结果是否与直接逐组求解一致。

最终这四类违规计数全部为 0，因此问题 3 的结果不仅“好看”，而且“合规并可复核”。

## 7. 输出文件说明

每个问题都会生成：

1. 一个综合工作簿 `problemX_tables.xlsx`
2. 若干单独 CSV / XLSX
3. 若干图形 PNG
4. 一个 `problemX_summary.json`

其中 JSON 用于快速读取关键结论，主论文里的多数数字都来自这些 JSON 和 CSV 文件。

对问题 2 而言，本轮新增和重点保留的关键产物包括：

1. `problem2_early_model_metrics`：5×5 repeated-CV 汇总指标。
2. `problem2_model_cv_detail`：5×5 repeated-CV 明细。
